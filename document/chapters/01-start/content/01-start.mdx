## 為什麼需要 OpenPress

AI 很適合產生草稿，但正式文件不只是文字。提案、白皮書、教學講義、研究筆記與產品手冊通常還需要固定版面、清楚章節、可追蹤 source、PDF 匯出、公開 reader 與後續維護。OpenPress 的定位，是把這些需求放進同一個文件 workspace。

OpenPress 不是「請 AI 幫我做一份檔案」的工具，而是一套 AI-first document framework。你可以讓 Claude Code、Cursor、Codex 或其他 agent 協助寫作與調整，但最後交付的是可重複產生的文件專案，而不是一次性的 Word、PDF 或 HTML。

### 和直接請 AI 產文件有什麼不同

<TableCaption>常見 AI 文件產生方式與 OpenPress 的差異</TableCaption>

| 做法 | 適合情境 | 常見限制 | OpenPress 的差異 |
| --- | --- | --- | --- |
| AI 產 Word | 快速整理文字、內部草稿 | 版面容易漂移，版本差異難審查 | source 是文字與元件，可 diff、可重建 |
| AI 產 PDF | 一次性簡報或短文件 | 產出後很難精修與延伸 | PDF 是 build artifact，不是唯一 source |
| AI 產 HTML | 需要快速展示頁面 | print、分頁、頁碼、封面與 TOC 不穩定 | 從一開始就以固定版面文件為核心 |
| Markdown 網站 | 技術文件與長文 | 不一定有 A4/PDF 契約 | 同時支援 web reader 與 PDF 交付 |

OpenPress 的重點不是取代 AI，而是替 AI 建立邊界。Agent 可以幫你寫內容、重排章節、補表格、改標題、調整風格；OpenPress 則負責把這些修改落在固定的 workspace、source model 與輸出流程中。

### OpenPress 的主要優勢

<TableCaption>OpenPress 對正式文件工作流的主要優勢</TableCaption>

| 優勢 | 對使用者的意義 |
| --- | --- |
| 固定 A4 版面 | 文件一開始就朝可列印、可匯出、可審查的形狀前進 |
| Source-first | 所有內容都能被 git、agent 與 reviewer 追蹤 |
| Web reader + PDF | 同一份 source 產生線上閱讀版本與正式 PDF |
| Style pack | 不同文件類型可以有自己的封面、目錄、字體與章節節奏 |
| Agent skills | Agent 知道要怎麼寫、怎麼改、怎麼部署，而不是每次重新解釋 |
| Deploy gate | 公開發布需要明確確認，避免 agent 誤發未完成內容 |

### 什麼時候該用 OpenPress

如果你只需要一段文字或一頁臨時 HTML，直接請 AI 產生就夠了。OpenPress 適合的是需要「反覆修改並交付」的文件：有章節、有封面、有圖表、有版本、有公開 URL，甚至需要多人或 agent 長期維護。

典型使用情境包括：

<TableCaption>OpenPress 適合的文件類型</TableCaption>

| 文件類型 | OpenPress 能幫上的地方 |
| --- | --- |
| 商業提案 | 固定封面、摘要、章節、圖表與 PDF 交付 |
| 白皮書 | 長文結構、事實邊界、引用與公開 reader |
| 教學講義 | 章節、練習題、答案附錄與可列印格式 |
| 產品手冊 | 可持續更新的 source 與部署流程 |
| 技術說明 | code block、表格、diagram 與版本化 |

第一章的重點很簡單：OpenPress 不是讓 AI 少寫，而是讓 AI 寫出的東西可以被維護、被輸出、被部署。
