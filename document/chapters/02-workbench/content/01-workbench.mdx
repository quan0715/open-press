## 開發者的兩條 Dev Loop

OpenPress framework 需要兩個不同的本機開發入口。第一個是 workspace dogfood document，用來驗證文件工作流；第二個是 public website，用來開發 `open-press.dev` 官網。兩者都重要，但不應混在同一個 dev command 裡。

### Dogfood Document

`pnpm run dev:workspace` 啟動 root `document/` 的 OpenPress Workbench。這是 framework 開發者最常用的驗證入口，因為它會走完整的 document source、theme、React reader、Project panel、PDF endpoint 與 deploy endpoint。

```bash
pnpm run dev:workspace
```

這個入口應回答三個問題：

| 問題 | 觀察位置 |
| --- | --- |
| 文件能不能被正確載入 | Document view |
| source 與 generated 邊界是否清楚 | Project view |
| 本機操作是否能觸發 PDF / deploy 狀態 | Workbench controls |

如果 framework 變更會影響 renderer、pagination、assets、comments、deploy status 或 media handling，就應該優先跑這條 loop。

### Public Website

`pnpm run dev:web` 啟動 `apps/web`。這是 open-press.dev 的產品官網，不是 document workspace。它負責對外介紹產品、引導使用者安裝 CLI、理解 style packs 與 agent 工作流。

```bash
pnpm run dev:web
```

官網可以連到公開 story，但不應直接承擔 dogfood document 的驗證責任。官網若壞了，代表 marketing / docs experience 有問題；dogfood document 若壞了，代表 framework pipeline 可能有問題。

### 預設 `dev`

Root `pnpm run dev` 應指向 `dev:workspace`。原因是 framework 開發的主要風險來自 document pipeline，而不是官網頁面。需要開發官網時，開發者明確使用 `dev:web`。

| Command | 目標 | 何時使用 |
| --- | --- | --- |
| `pnpm run dev` | root dogfood workspace | 預設 framework 開發 |
| `pnpm run dev:workspace` | root dogfood workspace | 明確檢查 document pipeline |
| `pnpm run dev:web` | `apps/web` 官網 | 修改 public landing site |

這個命名也讓 agent 比較不容易跑錯目標。當使用者說「跑 openpress 文件」時，應該啟動 workspace；當使用者說「跑官網」時，才啟動 web app。
