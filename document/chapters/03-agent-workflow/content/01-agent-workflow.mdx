## 編輯、調整與內容擴充

OpenPress 最自然的使用方式，是把 agent 當成寫作夥伴，而不是一次性產生器。你先建立文件骨架，再讓 agent 依照章節目標擴充內容、調整語氣、補表格、整理例子，最後用 preview 和 PDF 檢查版面是否真的成立。

### 從骨架開始

一份正式文件通常先需要三層結構：

<TableCaption>正式文件骨架的三個決策層級</TableCaption>

| 層級 | 問題 |
| --- | --- |
| 文件目標 | 這份文件要說服誰、交付什麼？ |
| 章節順序 | 讀者應該先理解背景、流程還是細節？ |
| 每章任務 | 這章負責解釋、比較、教學、說服還是驗證？ |

先讓 agent 幫你產生 outline，再決定哪些章節要深入。不要一開始就要求完整長文，因為固定版面文件需要考慮頁面節奏、表格高度、caption、章節長度與 PDF 輸出。

### 內容擴充 Prompt

當你要擴充某章，可以用這種 prompt：

```text
Use OpenPress writing conventions.

Expand chapter 2 into a user-facing guide.
Keep the tone practical and calm.
Use tables only when they help compare choices.
Do not invent product claims.
Mark missing facts as [TODO].
Keep each section short enough to fit fixed-layout pages.
```

這類 prompt 比「幫我寫完整第二章」更可靠。它明確告訴 agent：要面向誰、語氣如何、表格何時使用、事實不確定時怎麼處理，以及固定版面要避免過長段落。

### 調整內容時看三件事

<TableCaption>內容調整時的版面與交付檢查點</TableCaption>

| 檢查點 | 看什麼 |
| --- | --- |
| 讀者路徑 | 章節順序是否讓第一次接觸的人能理解？ |
| 版面節奏 | 是否有太長段落、孤立表格、頁尾被擠壓？ |
| 可交付性 | 這章放到 PDF 和 web reader 都合理嗎？ |

OpenPress 不要求你手動排每一頁，但它會讓版面問題提早出現。當內容太長、表格太密、標題層級太碎時，preview 會比純文字更早提醒你該重排。

### Workbench 的角色

在本機開發時，可以啟動 OpenPress Workbench：

```bash
pnpm run dev
```

Workbench 讓你看到文件、source、page state 與輸出入口。它不是單純的 Markdown preview，而是接近最終 reader/PDF 的工作視窗。當你請 agent 修改內容後，應該回到 Workbench 看實際頁面，而不是只讀 diff。

### 和 Agent 協作的節奏

一個健康的編輯循環通常是：

<TableCaption>OpenPress 與 agent 協作的基本循環</TableCaption>

| 步驟 | 動作 |
| --- | --- |
| 1 | 說明本章目標與讀者 |
| 2 | 讓 agent 改一小段或一個章節 |
| 3 | 在 reader preview 檢查文字與版面 |
| 4 | 要求 agent 收斂、補表格或重排 |
| 5 | 跑 validate/render/pdf 確認輸出 |

OpenPress 的重點是把 AI 協作變成可檢查的文件流程。每次改動都應該能回答：source 在哪裡、reader 長什麼樣、PDF 是否能產出。
