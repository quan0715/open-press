## 匯出與部署

OpenPress 的交付不是停在 preview。當內容穩定後，同一份 source 可以產出 web reader、PDF 與 deploy stage。這讓文件既能線上閱讀，也能被下載、審查或封存。

### 三種輸出

<TableCaption>OpenPress 三種主要輸出與用途</TableCaption>

| 輸出 | 用途 |
| --- | --- |
| export | 產生 reader 需要的 document model |
| render | 產生可部署的靜態 web reader |
| pdf | 產生固定版面 PDF |

這三種輸出對應不同檢查點。Export 確認內容模型成立；render 確認 web reader 能被建出來；PDF 確認固定版面和 print rules 沒壞。

### 常用命令

```bash
pnpm run build:doc        # validate + render the dogfood document
pnpm run openpress:pdf    # 固定版面 PDF
```

`build:doc` 結合了 validate(結構檢查)與 render(產出 `dist-react/`),適合在提交前驗證內容模型與 web reader 都成立;`pdf` 則確認最終可交付版本。若你只看瀏覽器 preview,很容易漏掉 PDF page break、字體或列印規則問題。如果只想跑結構檢查,可以直接呼叫 `node packages/core/engine/cli.mjs validate .`。

### 部署到 Cloudflare Pages

OpenPress workspace 可以設定 Cloudflare Pages deploy target。這份 User Story Book 的公開 reader 部署在：

```text
https://open-press-story.pages.dev
```

部署前應該先檢查 target，再由使用者明確確認：

```bash
pnpm run openpress:deploy:dry-run
pnpm run openpress:deploy -- --confirm
```

公開部署是有風險的動作，所以 OpenPress 會把 deploy 設計成需要確認的流程。Agent 不應在沒有使用者明確同意時發布文件。

### Web reader 和 PDF 應該一起看

<TableCaption>Web reader 與 PDF 的交付檢查差異</TableCaption>

| 檢查 | Web reader | PDF |
| --- | --- | --- |
| 章節導覽 | hash/page navigation | bookmarks/頁碼 |
| 閱讀體驗 | scroll、public URL | 列印與附件 |
| 版面穩定 | responsive shell | fixed page contract |
| 發布 | Cloudflare Pages | downloadable artifact |

OpenPress 的好處，是讓這兩種交付形式來自同一份 source。你不需要維護一份網頁和另一份 PDF，只需要維護 workspace，然後讓 pipeline 產生不同輸出。

### 什麼時候算可以公開

公開前至少檢查：

<TableCaption>公開部署前的最低檢查項目</TableCaption>

| 項目 | 問題 |
| --- | --- |
| 內容 | 是否還有 `[TODO]`、假資料或未確認敘述？ |
| Reader | 公開 URL 是否能開，章節是否能跳轉？ |
| PDF | 第一頁、目錄、頁碼、表格和圖是否正常？ |
| Deploy | project name 是否正確，是否需要 custom domain？ |

OpenPress 把文件交付變成一條可重複的流程：寫作、預覽、驗證、匯出、部署。這就是它和一次性 AI 文件產生最大的差異。
