`<Text>` provides a wrapper layer without built-in styles, used to encapsulate literal text into an `ObjectEntity` that supports system markup, inline editing, and commenting.

<div class="callout">
  <strong>Design Contract:</strong> This component does not provide any typographic properties (like font, size, color). Visual presentation must be implemented via `className` and external CSS expressions. Its sole purpose is to construct stable object identifiers.
</div>

  ### Example: Implicit Source Binding (Literal Text)

  ```tsx
  <Text as="h1" objectId="title" label="Slide title" className="slide-title">
    Fixed canvas workflow
  </Text>
  ```
  *The system will automatically inject file coordinates during SSR compilation to support editing tools and comments.*

  ### Example: Explicit Source Binding (Dynamic Variables)

  ```tsx
  <Text
    as="h1"
    objectId="title"
    label="Slide title"
    source={{
      path: "press/slide/press.tsx",
      kind: "tsx-text",
      objectId: "title",
      source: { line: 1, column: 16, endLine: 1, endColumn: 31 },
    }}
  >
    {dynamicTitle}
  </Text>
  ```

## Source Binding Behavior

- **Literal Node**: An `<Text>` instance imported from `@open-press/core` and internally containing only plain text. The engine automatically injects the `EditableSourceRef` during AST traversal.
- **Expression Node**: Content generated depending on variables, function calls, or array maps lacks automatic mapping capabilities. Developers must explicitly pass the `source` prop to unlock editing features.
- **Explicit Declaration Priority**: If a developer manually provides the `source` prop, the engine will adopt that value directly and bypass the automatic calculation mechanism.
