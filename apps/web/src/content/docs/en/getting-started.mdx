<div class="callout">
    <strong>Prerequisites.</strong> Node.js 20+ with <code>npm</code> and <code>npx</code>; use Node.js 24 for framework development and Cloudflare Pages builds.
    Chromium and <code>wrangler</code> are only needed when exporting PDF/images or deploying to Cloudflare.
  </div>

  <h2>1. Choose your start path</h2>

  <h3>Path A · AI-first: install skills, then ask the agent</h3>

  <p>
    Best for AI-first users, non-technical users, and opinionated formats. Install the OpenPress
    skill bundle first, then add any creative or domain skill that knows the publication style.
  </p>

  <h3>Example: Install skills</h3>

```bash
npx -y skills@latest add \\
  quan0715/open-press

# Optional creative/domain skill
npx -y skills@latest add \\
  quan0715/openpress-social-card-skill
```

  <p>
    Restart the agent session after installation so the new skills are visible. Then ask the agent
    to create the workspace. The OpenPress skills should guide it to run the create package, install the
    packages, copy or adapt skill-owned starter files, and verify the result.
  </p>

  <h3>Example: Prompt the agent</h3>

```text
我想做一份固定版面文件。請使用 OpenPress skills：
1. 確認 Node / npm / npx 可用
2. 用 npm create @open-press 建立 workspace
3. 安裝需要的 @open-press packages
4. 套用合適的 skill starter 或範例
5. 跑 npm run build 驗證
```

  <h3>Path B · CLI-first: run the commands yourself</h3>

  <p>
    Best for developers who already know the target folder, title, and workflow. The
    <code>npm create</code> command downloads the create package; it creates the workspace and runs
    <code>npm install</code> by default.
  </p>

  <h3>Example: Create workspace</h3>

```bash
npm create @open-press@latest my-deck -- \\
  --type slides \\
  --title "Transport models"

cd my-deck
```

  <h3>Example: If package install was skipped</h3>

```bash
npm install

# If skill sync failed
npm run openpress:skills
```

  <p>
    Starter content comes from skills, not bundled packs. OpenPress initializes the document
    workspace; skills own intake, examples, starter files, and taste. See
    <a href="/docs/skills">Skills</a> for the ownership split.
    For reports, proposals, books, or other page-based artifacts, use the AI-first path with
    <code>openpress-create-pages</code>; the create package only bootstraps the workspace shell.
  </p>

  <h2>2. Edit content</h2>

  <p>
    Everything you write lives under <code>press/</code>. Runtime internals live in
    <code>node_modules/@open-press/</code> after install; treat those packages as read-only.
  </p>

  <h3>Slides workspace</h3>

  <p>
    A slides Press uses a <strong>folder-per-slide</strong> layout. Each slide lives in its own
    directory under <code>slides/</code>; <code>press.tsx</code> is the ordered index that lists
    slides as JSX children. See <a href="/docs/concepts/slides">Slides architecture</a> for the full
    contract.
  </p>

  <h3>Example: Slides file tree</h3>

```text
press/<slug>/
  press.tsx          # ordered index — list slides in JSX order
  slides/
    intro/
      slide.tsx      # one slide: meta + default export component
    pricing/
      slide.tsx
  themes/
    default.css
```

  <h3>Example: slides/intro/slide.tsx</h3>

```tsx
import type { SlideMeta } from "@open-press/core";

export const meta = {
  layout: "default",
  description: "Opening slide",
} satisfies SlideMeta;

export default function Slide() {
  return <div>Write slide content in JSX.</div>;
}
```

  <h3>Example: press/`<slug>`/press.tsx (ordered index)</h3>

```tsx
import { Press, Slide } from "@open-press/core";

export default function DeckPress() {
  return (
    <Press slug="deck" title="My deck" type="slides" page="slide-16-9">
      <Slide id="intro" />
      <Slide id="pricing" />
    </Press>
  );
}
```

  <h3>Common to all press types</h3>

  <ul>
    <li><code>press/&lt;slug&gt;/themes/</code> — CSS tokens and theme overrides scoped to this press.</li>
    <li><code>press/&lt;slug&gt;/components/</code> — Press-local React components.</li>
    <li><code>press/&lt;slug&gt;/media/</code> and <code>press/shared/media/</code> — images, vector assets. Synced into the public bundle on export.</li>
    <li><code>package.json</code> — the <code>"openpress.deploy"</code> field is where build-time deploy adapter settings live. See <a href="/docs/concepts/workspace-config">Workspace config</a>.</li>
  </ul>

  <h2>3. Live preview</h2>

### Example: Start the workbench

```bash
npm run dev
# → http://127.0.0.1:5173/workspace
```

  <p>
    The workbench reloads CSS, tokens, and React chrome on save. MDX content updates flow through
    the engine's source watcher. The <code>/workspace</code> route opens the project gallery, and
    each document opens at <code>/&lt;press-slug&gt;/preview</code> with the workbench shell
    (inspector, source-edit endpoint, comment markers).
  </p>

  <h2>4. Build + verify</h2>

  <h3>Example: Production build</h3>

```bash
npm run build         # validates + renders dist-react/
npm run preview       # serve dist-react/ as a static site
npm run openpress:pdf # optional: generate PDF locally
```

  <p>
    <code>build</code> chains structural validation, MDX → React export, and Vite production build.
    If validation fails the build aborts before Vite runs, so a green build means the document
    shape is consistent. See <a href="/docs/concepts/cli-lifecycle">CLI · Lifecycle</a> for what each step
    does.
  </p>

  <h2>5. Deploy</h2>

  <p>
    OpenPress requires explicit confirmation for any deploy — there is no silent publish. Configure
    the adapter under the <code>"openpress.deploy"</code> field in the workspace
    <code>package.json</code>, then:
  </p>

  <h3>Example: Deploy to Cloudflare Pages</h3>

```bash
npm run openpress:deploy:dry-run        # preview the steps
npm run openpress:deploy -- --confirm   # publish
```

  <p>
    The deploy command builds, generates a PDF stage artifact, writes <code>deploy.json</code>
    metadata, and hands off to the configured adapter. If a host-specific adapter is not
    implemented, use the generated build output directly and keep the publish step outside
    OpenPress.
  </p>
