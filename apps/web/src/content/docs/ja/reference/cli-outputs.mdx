<p>
    ティア2コマンドは、同じエンジン出力の周りに重い依存関係（PDF/画像用のChromium、デプロイ用のSDKアダプター）をラップします。それぞれにランタイムコストがかかるため、これらは<code>build</code>のフラグではなく明示的なコマンドとして用意されています。<code>build</code>内で暗黙的に実行すると、HTMLバンドルだけを必要とするユーザーを驚かせることになるためです。
  </p>

    <h3>例: ワンショットPDF</h3>

```bash
npm run openpress:pdf
# → ./<config.pdf.filename>.pdf
```
  </ApiEntry>

  <ApiEntry
    name="openpress:deploy"
    kind="command"
    importFrom="npm run openpress:deploy"
    summary="設定されたデプロイアダプターを実行します。Workspaceをビルドし、PDFステージアーティファクトを並行して生成し、deploy.jsonにデプロイメタデータを書き込み、アダプターに引き渡します。常に--confirmを要求します。サイレントデプロイはありません。"
  >
    <PropsTable
      title="フラグ"
      rows={[
        {
          name: "--confirm",
          type: "flag",
          description:
            "実際のデプロイに必要です。対話型のプロンプトをスキップします。CIは別のゲート（例：PR承認）の後にのみこれを設定する必要があります。",
        },
        { name: "--dry-run", type: "flag", description: "プリフライトのみを実行します。コマンドシーケンス（ビルド → PDF → アダプター）を出力し、公開はしません。" },
      ]}
    />

    <h3>例: ローカルからのドライラン</h3>

```bash
npm run openpress:deploy:dry-run
# → アダプター、ターゲット、変更点を報告します
```

    <h3>例: 実際のデプロイ (CI)</h3>

```bash
npm run openpress:deploy -- --confirm
# → 1. ビルド
# → 2. デプロイステージへのPDF
# → 3. deploy.jsonを書き込み
# → 4. アダプターが公開
```

    <p>
      アクティブなアダプターは<code>package.json "openpress.deploy.adapter"</code>で選択されます。各アダプターには独自の必須設定があります。完全なスキーマについては<a href="/docs/concepts/workspace-config#operational-packagejson-openpress">Workspace設定</a>を参照してください。
    </p>
  </ApiEntry>

  <h2>アダプター</h2>

  <ApiEntry
    name='"cloudflare-pages"'
    kind="config"
    importFrom={`// package.json
"openpress": {
  "deploy": { "adapter": "cloudflare-pages", "projectName": "...", "source": ".deploy" }
}`}
    summary="npx wrangler pages deployを呼び出します。必須: projectName。オプション: commitDirty。デプロイするマシンでWranglerが認証されている必要があります。"
  />

  <p>
    他のホストは、生成されたビルド出力を直接消費できます。OpenPressはデプロイメントアダプターを明示的に維持します。ホスト固有のアダプターが実装されていない場合、エージェントは隠された公開パスを発明するのではなく、その境界を報告する必要があります。
  </p>
