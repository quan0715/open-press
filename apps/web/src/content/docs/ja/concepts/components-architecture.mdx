OpenPressのコアな哲学は、「ドキュメント」をReactのコンポーネントツリーとして扱うことです。これにより、Webページやアプリを開発するのと同じように、宣言的（Declarative）な構文を使ってドキュメントのコンテンツを構成できます。

OpenPressドキュメントを構成する3つの主なコンポーネント階層は以下の通りです：

## 1. <code>&lt;Workspace&gt;</code>

**Workspace** は最上位のコンテナであり、完全なプロジェクトまたはリポジトリを表します。
1つのワークスペースには、複数の異なる出版物（例：企画書とスライドデッキ、または同一ドキュメントの異なる言語バージョン）を含めることができます。

* **役割：** プロジェクトレベルの属性や共有設定を定義します。
* **配置：** 通常、ユーザーが手動で記述することはありません。エンジンが <code>press/</code> フォルダ構造を解析する際に、最上位のWorkspaceを自動的に生成します。

## 2. <code>&lt;Press&gt;</code>

**Press** は独立したドキュメント、プレゼンテーション、またはソーシャルメディアカードセットを表します。
フォルダの規約に従ったOpenPressプロジェクトでは、<code>press/&lt;slug&gt;/press.tsx</code> にある各ファイルは、単一の <code>&lt;Press&gt;</code> コンポーネントをデフォルトエクスポートする必要があります。

* **役割：** ドキュメントのタイトル、ページのジオメトリ（A4や16:9スライドなど）、およびコンテンツソース（MDXソース）を宣言します。
* **内部動作：** Pressは内部のページネーション状態を管理します。PDFエクスポート時、エンジンはこのツリーを複数回レンダリングして、ページ数やコンテンツの流れを正確に計算します。

## 3. <code>&lt;Frame&gt;</code>

**Frame** は、Press内の物理的なページ表面（Page Surface）を表します。
Pressが本全体だとすれば、Frameは表紙、裏表紙、目次ページ、あるいはコンテンツが連続して流れる本文領域になります。

* **役割：** コンテンツを配置できる境界を提供します。
* **種類：**
  * **単一ページFrame：** 表紙や裏表紙のように、コンテンツが固定されておりページをまたがないもの。
  * **流動コンテンツ：** <code>&lt;Sections&gt;</code> のように、MDXの長さに応じて内部的に自動で複数の物理ページにまたがるもの。

## 例：コンポーネントツリーの構造

```tsx
<Workspace name="2026-report">
  {/* 1つ目のPress：A4レポート */}
  <Press slug="report" title="年次レポート" page="a4" sources={[...]}>
    <Frame frameKey="cover"><Cover /></Frame>
    <Toc source="story" />
    <Sections source="story" /> {/* コンテンツの長さに応じて自動的に複数ページに展開 */}
    <Frame frameKey="back-cover"><BackCover /></Frame>
  </Press>

  {/* 2つ目のPress：16:9 プレゼンテーションスライド */}
  <Press slug="slides" title="プレゼンテーション" page="slide-16-9" sources={[...]}>
    <Slide id="intro" />
    <Slide id="metrics" />
  </Press>
</Workspace>
```

## まとめ

- 従来のWYSIWYGエディターの代わりに **コンポーネントツリー** を使用することで、**Agent** が正確なJSX構文を通じてレイアウトと構造を制御できるようになります。
- <code>&lt;MdxArea&gt;</code> のような高度なコンポーネントは、MDXテキストをReactコンポーネントにマッピングする役割を担います。
- 具体的なAPIパラメータについては、左側の **リファレンス (Reference)** セクションを参照してください。
