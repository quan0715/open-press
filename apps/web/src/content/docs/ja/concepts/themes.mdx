OpenPress において、テーマ（Themes）の設計はウェブスタイルの本質に立ち返っています。つまり、純粋な CSS とデザイントークン（Design Tokens）です。私たちは、複雑な JavaScript のテーマランタイムや巨大な設定オブジェクトを意図的に避けています。

## なぜテーマはただの CSS なのか？

現代の多くのフレームワークはスタイルロジックを JavaScript 内にカプセル化する傾向がありますが、OpenPress は異なる道を選びました：

1. **唯一の信頼できる情報源：** すべての色、フォント、余白の基礎として純粋な CSS 変数（トークン）を使用することで、ドキュメント全体で視覚設定の一貫性を確保します。組み込みのものであれカスタムのものであれ、どのコンポーネントもこれらのネイティブなスタイルの基盤をシームレスに利用できます。
2. **ページジオメトリとの深い統合：** 物理ドキュメント（A4 レポートやプレゼンテーションなど）の設計は、正確な物理的寸法に大きく依存しています。OpenPress エンジンはビルド時に、ページの幅、高さ、アスペクト比を動的に CSS 変数として注入します。これにより、テーマは自身が配置されている物理的空間を認識し、レスポンシブまたはアダプティブなタイポグラフィを実装できるようになります。
3. **明確な関心の分離：** テーマは「どのように見えるか」（視覚設計）に焦点を当て、React コンポーネント（表紙や目次など）は「構造は何か」（マークアップ言語）に焦点を当てます。この分離により、デザインの置き換えが非常に軽量になります。

## コア設計コンセプト

OpenPress のテーマを理解するには、いくつかの重要な抽象化のレイヤーを把握する必要があります：

### デザイントークン (Tokens)

トークンはテーマを構成する原子です。これらにはタイポグラフィのロジックは含まれておらず、特定の色やフォントファミリーに対する単なる「命名」に過ぎません。例えば、特定の青色のカラーコードをあちこちにハードコーディングするのではなく、「アクセントカラー」というトークンを定義します。これにより、包括的なブランドの更新も一箇所の変更だけで済みます。

### ページ契約とジオメトリ

物理メディア（PDF など）にエクスポートする必要があるドキュメントの場合、テーマは「ページ契約」を定義する必要があります。これには、印刷可能な物理的領域やマージンの定義、コンテンツが境界線を越えないようにすることなどが含まれます。エンジンによって注入されたジオメトリ変数はここで重要な役割を果たし、同じスタイルを異なる用紙サイズにインテリジェントに適応させることができます。

### タイポグラフィの比率と基本ルール

この部分は、ドキュメント内で最も一般的なコンテンツ要素（見出し、段落、リストなど）を処理します。ドキュメントの基本的なリズムを確立し、誰がコンテンツを作成しても、一貫した優れた読書体験のタイポグラフィ比率を確実に提供できるようにします。

### 進化と互換性

OpenPress は、開発者が React と Tailwind を通じてカスタムページ UI を構築するための強力なメカニズムを提供していますが、テーマシステムは依然として従来のスタイルパターンの互換性を保持しています。この設計により、既存のドキュメントがスムーズに移行できるよう保証しつつ、より現代的でコンポーネント化された設計アプローチへの扉を開いています。
