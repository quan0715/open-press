<p>
    OpenPress 註解並不是獨立的服務。每一個註解都是直接寫入原始碼檔案中的 MDX 註解區塊 — workbench 會在渲染時讀取它們，並將其顯示為互動式圖釘。這意味著註解會與文件一起在 git 中留存，能夠跨分支存在，並且可以被任何能編輯 MDX 的工具（包括 AI agents）所處理。
  </p>

  <div class="callout">
    <strong>權衡與註解資料庫的差異。</strong> 刻意設計成沒有帳號、沒有通知、沒有討論串回覆。如果您需要這些功能，請將部署好的閱讀器放在如 Hypothes.is 這樣的服務後方，或是自行建立註解後端。OpenPress 只負責行內標記。
  </div>

    <h3>範例：在 MDX 內文中</h3>

```mdx
The model assumes uniform demand across the network — see
chapter 3 for the relaxation under sparse-link conditions.

{/* @openpress-comment id="c-2026-05-28-uniform-demand" ts="2026-05-28T12:34:00Z" text="Is 'uniform' the right word here? Maybe 'isotropic'?" author="reviewer" */}
```

    <p>
      多個標記可以放在同一段內容旁邊。workbench 會將它們群組在同一個區塊上，但渲染為獨立的圖釘，使每個註解保持可被定址的狀態。原始碼中的位置決定了面板中的視覺順序。
    </p>
  </ApiEntry>

  <h2>撰寫註解</h2>

  <h3>從 workbench（人類）</h3>

  <ol>
    <li>執行 <code>npm run dev</code>，開啟 <code>http://127.0.0.1:5173/workspace</code>。</li>
    <li>從工具列開啟 inspector。</li>
    <li>點擊一個區塊（或區塊間的插入條）→ Add comment。</li>
    <li>輸入註解，按 <code>Cmd/Ctrl + Enter</code> 送出。</li>
    <li>workbench 會將標記寫回 MDX 原始碼並儲存檔案。</li>
  </ol>

  <h3>從 agent（Claude / Codex / Cursor）</h3>

  <ol>
    <li>使用 <code>open-press search . "&lt;query&gt;" --json</code> 搜尋原始碼檔案以定位區塊。</li>
    <li>直接在目標文字後方插入標記行。使用全新的 <code>id</code>（類似 UUID — 慣例為 <code>c-YYYY-MM-DD-&lt;slug&gt;</code>）和目前的 ISO 時間戳記。</li>
    <li>使用 <code>open-press replace . "&lt;old&gt;" "&lt;new&gt;" --apply</code> 或是直接使用編輯器的 MDX 編輯工具寫入檔案。</li>
  </ol>

  <h2>處理註解</h2>

  <p>
    註解是文件的行內待辦事項。工作流程為 <strong>閱讀 → 決定 → 編輯原始碼 → 移除標記</strong>。移除標記是一個積極的動作，表示「這已經處理好了」 — 絕不要在沒有套用變更（或記錄跳過的決定）的情況下刪除標記。
  </p>

  <h3>範例：尋找所有待處理的標記</h3>

```bash
rg "@openpress-comment" press/ -n
```

  <p>
    Agents 有專門的工作流程來處理這個 — 請參見 <a href="/docs/reference/cli-tools">CLI 工具</a> 頁面以及內建的 <code>openpress-apply-comments</code> skill。該 skill 的契約為：
  </p>

  <ol>
    <li>列出所有待處理的標記。</li>
    <li>針對每個標記，閱讀周圍的文字，決定要進行的原始碼編輯，並套用它。</li>
    <li>只有在編輯完成後才刪除標記。</li>
    <li>執行 <code>npm run build</code> 驗證沒有任何東西壞掉。</li>
    <li>回報哪些標記已解決，以及哪些（如果有的話）因為需要人類做決定而被保留。</li>
  </ol>

  <h2>反模式 (Anti-patterns)</h2>

  <ul>
    <li>
      <strong>不要跨文件重複使用 ids。</strong> <code>id</code> 是 workbench 圖釘的定址鍵 — 重複的 id 會破壞導航。如果您分叉了一個章節且需要保留標記，請重新產生 id。
    </li>
    <li>
      <strong>不要在沒有執行任何動作的情況下清除標記。</strong> 標記是承諾。您必須套用變更，或是用後續的 TODO 註解替換標記來解釋為何要擱置它。
    </li>
    <li>
      <strong>不要手寫編碼過的 <code>text</code>。</strong> 使用 workbench，或是讓 agent 產生它 — 手動跳脫引號 / 換行很容易出錯。
    </li>
  </ul>
