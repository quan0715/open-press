在 OpenPress 中，佈景主題（Themes）的設計回歸到了網頁樣式的本質：純粹的 CSS 與設計代幣（Design Tokens）。我們刻意避免了複雜的 JavaScript 佈景主題執行期或是龐大的設定物件。

## 為什麼佈景主題只是 CSS？

許多現代框架傾向於將樣式邏輯封裝在 JavaScript 中，但 OpenPress 選擇了一條不同的路：

1. **唯一的真實來源：** 透過純 CSS 變數（Tokens）作為所有顏色的、字型與間距的基礎，我們確保了整個文件中視覺設定的一致性。任何元件，無論是內建的或是自訂的，都能無縫地取用這些原生的樣式基礎。
2. **與頁面幾何的深度整合：** 實體文件的設計（如 A4 報告或簡報）高度依賴精確的物理尺寸。OpenPress 引擎在建置時，會動態地將頁面的寬度、高度與比例注入為 CSS 變數。這使得佈景主題能夠感知它所處的物理空間，進而實作響應式或自適應的排版。
3. **清晰的關注點分離：** 佈景主題專注於「看起來如何」（視覺設計），而 React 元件（如封面、目錄）則專注於「結構是什麼」（標記語言）。這種分離讓設計的替換變得極為輕量。

## 核心設計概念

理解 OpenPress 佈景主題，需要掌握幾個關鍵的抽象層次：

### 設計代幣 (Tokens)

代幣是構成佈景主題的原子。它們不包含任何排版邏輯，僅僅是對於特定顏色值或字型家族的「命名」。例如，與其在各處寫死特定的藍色色碼，不如定義一個「強調色」代幣。這使得全面性的品牌更新只需要修改一個地方。

### 頁面契約與幾何

對於需要匯出為實體媒介（如 PDF）的文件，佈景主題必須定義「頁面契約」。這包含了定義可列印的物理區域、邊距，以及確保內容不會超出邊界。引擎注入的幾何變數在這裡扮演了關鍵角色，讓同一套樣式可以智慧地適應不同尺寸的紙張。

### 排版比例與基礎規則

這部分處理了文件中最常見的內容元素（如標題、段落、清單）。它建立了文件的基礎節奏，確保無論內容由誰編寫，都能呈現出一致且具備良好閱讀體驗的排版比例。

### 演進與相容性

雖然 OpenPress 提供了強大的機制讓開發者透過 React 與 Tailwind 來建立自訂的頁面 UI，但佈景主題系統仍保留了對傳統樣式模式的相容性。這種設計確保了既有文件能夠平滑地轉移，同時也為更現代、更元件化的設計方法敞開了大門。
