    <h3>範例：帶有自動來源對應的簡報文字</h3>

```tsx
import { Frame, Text } from "@open-press/core";

export function Slide() {
  return (
    <Frame frameKey="slide-01" role="canvas.slide" chrome={false}>
      <Text as="h1" objectId="title" label="Slide title" className="slide-title">
        Fixed canvas workflow
      </Text>
      <Text as="p" objectId="lede" label="Slide lede" className="slide-lede">
        Build slides as fixed Frames, then let the workbench handle comments and inline edits.
      </Text>
    </Frame>
  );
}
```

    <p>
      對於上面的兩個文字節點，匯出的 HTML 會自動接收 <code>data-openpress-object-source</code>。如此一來，workbench 就可以在 <code>press/&lt;slug&gt;/press.tsx</code> 中編輯確切的字面內容，而不需要作者或 agent 手動填寫行號。
    </p>

    <h3>範例：如果背後是表達式的文字需要可被編輯，則需提供明確的來源</h3>

```tsx
const title = "Generated title";

<Text
  as="h1"
  objectId="title"
  label="Slide title"
  source={{
    path: "press/slide/press.tsx",
    kind: "tsx-text",
    objectId: "title",
    source: { line: 1, column: 16, endLine: 1, endColumn: 31 },
  }}
>
  {title}
</Text>
```

  <h2>來源對應規則 (Source mapping rules)</h2>

  <ul>
    <li>
      <strong>字面上的子元素預設是可被編輯的。</strong> 如果 <code>&lt;Text&gt;</code> 是從 <code>@open-press/core</code> 匯入的，且其子元素是單一的字面文字節點，OpenPress 會在 React SSR 匯出期間推導出其原始碼範圍。
    </li>
    <li>
      <strong>不會猜測表達式的子元素。</strong> <code>{`{title}`}</code>、<code>{`{children}`}</code>、陣列以及透過 map 產生的內容，作為渲染物件仍然可以接受註解，但除非您提供 <code>source</code>，否則它們無法進行行內編輯。
    </li>
    <li>
      <strong>手動設定的來源優先。</strong> 如果提供了 <code>source</code> prop，OpenPress 會保留它，而不會另外注入一個新的。
    </li>
    <li>
      <strong>只有 OpenPress 的 Text 會被轉換。</strong> 名稱叫 <code>Text</code> 的局部元件會被忽略，除非它們是作為 <code>Text</code> 或是 <code>@open-press/core</code> 的別名被匯入。
    </li>
  </ul>

  <h2>何時該使用它</h2>

  <ul>
    <li>需要在 workbench 中直接編輯的簡報、卡片、封面以及封底文案。</li>
    <li>當客製化元件內部的文字是在 TSX 中撰寫時，其中的圖說或是精簡的標籤。</li>
    <li>畫布風格的頁面，在這種情況下不需要使用 MDX，且每個頁面都是固定的 React 元件。</li>
  </ul>

  <h2>何時不該使用它</h2>

  <ul>
    <li>已經透過 <code>&lt;MdxArea&gt;</code> 渲染的長篇散文；MDX 區塊已經帶有原始碼 metadata。</li>
    <li>與視覺樣式相關的考量。將排版與佈局放在 theme CSS 中，而不是 <code>Text</code> props 中。</li>
    <li>產生的字串，且其來源無法表示為單一個穩定可編輯範圍的內容。</li>
  </ul>
