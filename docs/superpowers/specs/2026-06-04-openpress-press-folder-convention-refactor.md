# OpenPress Press Folder Convention Refactor Spec

## Summary

OpenPress should move from a root-entry composition model to a filesystem-first Press folder convention.

The next version should treat every user artifact as a self-contained folder under `press/<folder>/`. Each folder owns its own `press.tsx`, local components, theme, and media. The engine discovers those Press folders, loads each folder's exported Press component, and builds the Workspace manifest from that collection.

This refactor is motivated by the create-pages / create-slide skill split and by dogfood from a generated slide project. The current generated slide output works, but it tends to produce props-heavy template DSLs such as `items`, `metrics`, `steps`, `pageNumber`, and `totalPages`. That shape is fast for first generation but weak for OpenPress's long-term source editing, comment targeting, and agent follow-up work.

The new slide authoring model should be closer to shadcn-style copied project components: minimal props, explicit JSX children, local ownership, and easy editing by humans or agents.

Slide source should use clear project folders:

- `components/` for deck-level wrappers such as `DeckSlide`.
- `ui/` for reusable content primitives that do not know about the slide canvas.
- `layouts/` for full-slide layout components that wrap `DeckSlide`.

Use "template" as skill and generation vocabulary, not as the default source folder name.

## Goals

- Make `press/<folder>/` the unit of authoring, editing, export, and agent reasoning.
- Let `openpress init <press-name> --type pages|slides` scaffold a single Press folder.
- Limit supported Press types to `pages` and `slides`.
- Remove root Press entry support; only `press/*/press.tsx` is recognized.
- Keep shared resources possible through `press/shared/`, but make shared resources optional rather than the default owner of all theme and media.
- Make slide generation produce React source that is directly readable and editable, not a props DSL.
- Provide a thin core `Slide` primitive that wraps `Frame` for slide semantics.
- Keep design-heavy slide components project-owned and generated by `openpress-create-slide`, not hidden inside core.
- Make standard slide layouts expose Text-backed compound slots for visible titles, descriptions, captions, and labels.

## Non-Goals

- Do not port the `open-slide` runtime into OpenPress.
- Do not add a third Press type such as `social`; social-card outputs remain either `pages` or a future separate decision.
- Do not design a `Deck` abstraction in this version.
- Do not add a root ordering override file in this version.
- Do not create a large slide component library inside `@open-press/core`.
- Do not make slide layouts accept large data props such as `items`, `metrics`, or `steps` as the primary authoring style.

## Current Problems

### Root Entry Owns Too Much

The previous multi-Press model relied on a root Press entry to mount a `Workspace` and compose every `Press`. In a generated project, that often became:

```tsx
import { Press, Workspace } from "@open-press/core";
import { config, qjudgeScrumFeatureReviewFrames } from "./qjudge-scrum-feature-review";

export default function OpenPressWorkspace() {
  return (
    <Workspace name="QJudge Scrum Feature Review">
      <Press slug="qjudge-scrum-feature-review" title={config.title} type="slides" page={config.page}>
        {qjudgeScrumFeatureReviewFrames()}
      </Press>
    </Workspace>
  );
}
```

This kept the real Press metadata outside the folder that owns the deck. The deck folder exported frames and config, while the root entry remained the actual Press registry. That made the root file a coordination bottleneck and forced create skills to modify shared project state even when adding one artifact.

### Generated Slide Layouts Become Data DSLs

The generated slide project uses layout components like:

```tsx
<AgendaSlide
  pageNumber={2}
  totalPages={totalPages}
  items={[
    { number: "01", title: "題目介紹", note: "交代 QJudge feature 的情境、使用者與報告範圍。" },
    { number: "02", title: "敏捷會議計劃", note: "規劃會議節奏、角色分工與每次討論的產出。" },
  ]}
/>
```

This creates several issues:

- The visible slide content is stored inside object arrays instead of slide JSX.
- Follow-up edits need to modify data objects rather than the exact rendered structure.
- `items.map()` output is weaker for source mapping and comment targeting.
- Layout components tend to accumulate more props over time.
- Page numbers are hardcoded into props, so slide insertion or reordering causes unrelated edits.

OpenPress should bias toward explicit source because source is the durable collaboration surface.

## Reference: open-slide

The `1weiho/open-slide` project is useful as a reference for slide authoring philosophy:

- A slide page is arbitrary React code.
- The canvas is fixed at 1920 x 1080.
- Agent rules are shipped with the scaffold.
- A slide has a local folder and local assets.
- Heading hierarchy is mostly governed by authoring rules and visual type scale, not by a formal `H1` / `H2` / `H3` component API.

OpenPress should adopt the "slides are React source, not a constrained DSL" principle. It should not copy open-slide's one-large-file rule. OpenPress needs long-term document maintenance, comments, source editing, PDF/image export, and folder-level ownership, so multi-file Press folders are a better fit.

For OpenPress slides, typography should be expressed through layout and text roles rather than raw heading levels. Agents should think in terms of hero title, section title, page title, card title, body, caption, and label. The final HTML tag can still be `h1`, `h2`, `h3`, `p`, or `span`, but the authoring API should expose the slide role first.

## Proposed Project Layout

### Default Workspace

```text
press/
├── shared/
│   ├── data.ts
│   ├── components/
│   ├── media/
│   └── theme/
├── report/
│   ├── press.tsx
│   ├── chapters/
│   ├── components/
│   ├── theme/
│   └── media/
└── slide/
    ├── press.tsx
    ├── components/
    ├── ui/
    ├── layouts/
    ├── theme/
    └── media/
```

`press/shared/` is optional. It exists only when multiple Press folders need common assets, shared facts, brand tokens, or reusable components.

Each Press folder is the default source of truth for its own artifact. In the common case, an agent can add or modify a Press without touching sibling folders.

### Required Per-Press Files

Every `press/<folder>/` must contain:

```text
press/<folder>/
├── press.tsx
├── components/
├── theme/
└── media/
```

Additional required files depend on Press type:

```text
press/<folder>/chapters/     # required for type="pages"
press/<folder>/ui/           # required for type="slides"
press/<folder>/layouts/      # required for type="slides"
```

The folder name is the default slug. The `Press` component may omit `slug` only if the engine injects or validates it from the folder name. The preferred explicit form is:

```tsx
<Press slug="slide" title="..." type="slides" page="slide-16-9">
  ...
</Press>
```

Component and media lookup is path-declared, not locked to a single fixed directory. Defaults include folder-local `./components` / `./media` and optional `press/shared/*`. If a Press intentionally uses another source folder, `<Press componentsDir>` and `<Press mediaDir>` accept a string or string array. `./` paths resolve relative to the owning Press folder; bare paths resolve relative to `press/`.

## Filesystem Discovery

The engine should discover Press entries by scanning:

```text
press/*/press.tsx
```

Only one convention should be recognized in this version. Do not also support `index.tsx`, `deck.tsx`, `document.tsx`, or `page.tsx` as equivalent Press entries.

Discovery rules:

- Ignore `press/shared/`.
- Ignore dot folders and generated output.
- Sort folders by stable filesystem name for deterministic output.
- Load each `press/<folder>/press.tsx` default export.
- The default export must be an inert component that returns one `<Press>`.
- The folder name becomes the default slug and must match the explicit `slug` prop if one is present.
- The engine builds a Workspace internally from discovered Press entries.

The root Press entry is unsupported in this version. Ordering is not a first-class concern yet.

## CLI Init Contract

`openpress init` should become folder-aware:

```bash
openpress init report --type pages
openpress init slide --type slides
```

The positional argument is the Press name. It seeds the folder name, slug, and default component name:

| Input | Folder | Slug | Component |
| --- | --- | --- | --- |
| `report` | `press/report/` | `report` | `ReportPress` |
| `slide` | `press/slide/` | `slide` | `SlidePress` |

Behavior:

- Create the target workspace if it does not exist.
- Create `press/<folder>/` if it does not exist.
- Scaffold the required folder structure for the requested type.
- Refuse to overwrite an existing non-empty Press folder unless an explicit future flag is added.
- Keep lifecycle operations such as upgrade and migrate under the general `openpress` workflow.

Supported `--type` values:

```text
pages
slides
```

There is no `social` Press type in this version.

## Pages Folder Contract

Default pages layout:

```text
press/report/
├── press.tsx
├── chapters/
├── components/
├── theme/
└── media/
```

Default `press.tsx` shape:

```tsx
import { Press } from "@open-press/core";
import { mdxSource } from "@open-press/core/mdx";
import { Sections, Toc } from "@open-press/core/manuscript";

export default function ReportPress() {
  return (
    <Press
      slug="report"
      title="Report"
      type="pages"
      page="a4"
      sources={[
        mdxSource({ id: "report", preset: "section-folders", root: "report/chapters" }),
      ]}
    >
      <Toc source="report" maxLevel={2} />
      <Sections source="report" />
    </Press>
  );
}
```

Pages continue to use MDX source registration. The create-pages skill owns cover, back-cover, chapter opener, TOC depth, theme, and manuscript structure.

There is no `Deck` abstraction in this version. The Press entry composes slides directly.

### Slide Identity

Slide identity must be stable and semantic. Do not use order-carrying keys such as `slide-01`, `slide-02`, or `page-03` as the default generated form.

Preferred:

```tsx
<TitleSlide id="cover" title="Product Review">
  <TitleSlide.Title objectId="title">Product Review</TitleSlide.Title>
</TitleSlide>

<TitledContentSlide id="problem-context" title="Problem Context">
  <TitledContentSlide.Title objectId="title">Problem Context</TitledContentSlide.Title>
  <TitledContentSlide.Content>
    <Text as="p" objectId="summary">The source content stays here.</Text>
  </TitledContentSlide.Content>
</TitledContentSlide>
```

Reordering slides should require moving JSX, not renumbering every key. Inserting a new slide should only add a new semantic id:

```tsx
<TitledContentSlide id="demo-flow" title="Demo Flow">
  <TitledContentSlide.Title objectId="title">Demo Flow</TitledContentSlide.Title>
  <TitledContentSlide.Content>
    <Text as="p" objectId="summary">Describe the new flow.</Text>
  </TitledContentSlide.Content>
</TitledContentSlide>
```

The core `Slide` component maps `id` to `Frame`'s `frameKey`. `frameKey` remains the engine-level identity, but author-facing slide APIs should use `id`.

Generated `press.tsx` should compose the actual deck content inline inside layout components. Do not generate one empty component per slide just to hide slide content behind `<ProblemContextSlide />` or `<Slide03 />`. Extract a new component only when it is a reusable `ui/*` primitive or a reusable `layouts/*` pattern.

## Core `Slide` Primitive

Core should provide a thin `Slide` component that wraps `Frame`:

```tsx
import { Slide } from "@open-press/core";

<Slide id="agenda" title="Agenda">
  ...
</Slide>
```

Expected behavior:

```tsx
<Frame
  frameKey={id}
  role="canvas.slide"
  chrome={false}
  data-page-title={title}
>
  {children}
</Frame>
```

`Slide` should not own visual design. It should not include theme, headers, footers, grids, logos, or layout opinions.

Minimal props:

- `id`
- `title?`
- `role?`, defaulting to `canvas.slide`
- `className?`
- `children`

The purpose is to prevent agents from repeatedly remembering the `Frame` slide contract while keeping all design choices in project-owned files.

## Shadcn-Style Slide Components

`openpress-create-slide` should generate project-owned components that the user can edit directly. These components should feel like shadcn components: copied into the project, explicit, and easy to modify.

Recommended local component responsibilities:

| Component | Responsibility |
| --- | --- |
| `DeckSlide` | The deck-local blank slide. Wraps core `Slide` and owns watermark, fixed chrome, footer, `PageFolio`, brand marks, and safe area. |
| `ui/*` | Reusable content primitives. They do not wrap `DeckSlide` and should not know about slide order, folio, or deck chrome. |
| `layouts/*` | Full-slide layout components that extend `DeckSlide` with slots and fixed structures for common slide roles. |

`DeckSlide` corresponds to a blank slide in presentation software, but with this deck's recurring chrome. It is intentionally project-owned because watermark, logo, footer copy, and visual rhythm are deck-specific.

`ui/` and `layouts/` are the default slide source folders:

- `ui/` contains reusable slide content primitives, such as `Text`, `Card`, `Badge`, `Callout`, `KpiCard`, `ImageFrame`, `QuoteBlock`, `Timeline`, and `CompareTable`. These components may render editable `Text`, but they should not wrap `DeckSlide` or own page-level chrome.
- `layouts/` contains slide-level components, such as `TitleSlide`, `AgendaSlide`, `ChapterOpenerSlide`, `TitledContentSlide`, `TwoColumnSlide`, `MetricSlide`, `ComparisonSlide`, `ImageCaptionSlide`, and `ConclusionSlide`. These components wrap `DeckSlide`, define fixed safe areas, expose slots, and may use `ui/*` primitives internally.
- "template" remains a useful word for skill intent: generating a reusable pattern, converting a sketch into a layout, or revising a starter. In project source, the default folder is `layouts/`.

Default `DeckSlide` shape:

```tsx
import { PageFolio, Slide } from "@open-press/core";
import type { ReactNode } from "react";

export function DeckSlide({
  id,
  title,
  variant = "default",
  children,
}: {
  id: string;
  title?: string;
  variant?: "default" | "cover" | "chapter" | "closing";
  children: ReactNode;
}) {
  return (
    <Slide id={id} title={title} className={`deck-slide deck-slide--${variant}`}>
      <div className="deck-slide__watermark" aria-hidden="true">OpenPress</div>
      <main className="deck-slide__main">{children}</main>
      <footer className="deck-slide__footer">
        <PageFolio variant="slash" currentFormat="2-digit" totalFormat="2-digit" />
      </footer>
    </Slide>
  );
}
```

`SlideHeader`, `SlideFooter`, and `SlideSection` are not standard scaffold files. They may be extracted later if a deck has complex chrome or repeated spacing needs. The default scaffold should start with one editable `DeckSlide.tsx`.

## Slide Layout Style

Slide layouts should prefer children composition over data props.

Preferred:

```tsx
<AgendaSlide id="agenda" title="本次報告結構">
  <section className="agenda-heading">
    <Text as="p" objectId="agenda-eyebrow">Agenda</Text>
    <Text as="h2" objectId="agenda-title">本次報告結構</Text>
  </section>

  <ol className="agenda-list">
    <li>
      <span>01</span>
      <div>
        <Text as="h3" objectId="agenda-01-title">題目介紹</Text>
        <Text as="p" objectId="agenda-01-note">交代情境、使用者與報告範圍。</Text>
      </div>
    </li>
  </ol>
</AgendaSlide>
```

Avoid:

```tsx
<AgendaSlide
  items={[
    { number: "01", title: "題目介紹", note: "交代情境、使用者與報告範圍。" },
  ]}
/>
```

Allowed props should be structural, not content-heavy:

- `id`
- `title?` for `data-page-title` and navigation metadata
- `chapter?` for chapter identity metadata when needed
- `variant?` for explicit layout variants
- `className?`
- `children`

Visible title, eyebrow, description, lead, caption, and label text should prefer Text-backed compound slots over string props. If a layout accepts a visible string prop for compatibility, it must render that string through `Text` internally with a predictable object id.

Avoid defaulting to props such as:

- `items`
- `metrics`
- `steps`
- `blocks`
- `pageNumber`
- `totalPages`
- `logo`
- `footerLabel`
- `showFolio`

If a slide genuinely needs repeated data, write the repeated JSX in the slide source first. Introduce a small local helper only when repetition is high and the helper improves readability without hiding source ownership.

## Slide Text Roles And Compound Slots

Standard slide layouts should expose compound components for formal text slots. These slots are the authoring API for editable slide text.

Recommended slot components:

| Slot | Visual role | Internal behavior |
| --- | --- | --- |
| `TitleSlide.Title` | hero title | wraps `Text` with the cover/hero title class |
| `TitledContentSlide.Eyebrow` | label or section marker | wraps `Text` with the eyebrow class |
| `TitledContentSlide.Title` | page title | wraps `Text` with the page title class |
| `TitledContentSlide.Description` | lead or description | wraps `Text` with the lead/body class; `Desc` may be provided as a short alias |
| `ChapterOpenerSlide.Chapter` | chapter marker | wraps `Text` with the chapter label class |
| `ChapterOpenerSlide.Title` | section title | wraps `Text` with the section title class |
| `ChapterOpenerSlide.Lead` | section lead | wraps `Text` with the lead/body class |
| `ChapterOpenerSlide.SectionList` | fixed opener section-list region | owns the list container, spacing, dividers, and alignment |
| `ChapterOpenerSlide.ListItem` | fixed marker + title + optional description row | wraps visible item text through `Text` with ids derived from the item id |
| `QuoteSlide.Quote` | quote text | wraps `Text` with the quote class |
| `QuoteSlide.Attribution` | caption | wraps `Text` with the caption class |

Example:

```tsx
<TitledContentSlide id="workflow" title="從題目到 Demo 的工作流程">
  <TitledContentSlide.Eyebrow objectId="eyebrow">
    Development Process
  </TitledContentSlide.Eyebrow>

  <TitledContentSlide.Title objectId="title">
    從題目到 Demo 的工作流程
  </TitledContentSlide.Title>

  <TitledContentSlide.Description objectId="description">
    說明從需求理解、拆解、實作到展示的工作節奏。
  </TitledContentSlide.Description>

  <TitledContentSlide.Content>
    <ProcessMap />
  </TitledContentSlide.Content>
</TitledContentSlide>
```

For one-per-slide formal slots, generated layouts may default `objectId` values such as `title`, `eyebrow`, `description`, `chapter`, or `lead` because they are scoped by the slide id. Repeated text, such as cards, process steps, table cells, annotations, and chart labels, should provide explicit semantic object ids.

This keeps layout usage ergonomic while still making inline edit and source mapping reliable. Plain JSX text is allowed for decorative chrome, but visible user-editable content should use `Text` or a local component that wraps `Text`.

For source-backed compound slots, the layout implementation should accept `TextProps` and pass them through to `Text`. The Press usage should provide `objectId` on visible literal text slots; the source transform can then inject source ranges for both direct `Text` usage and `objectId`-bearing compound slots.

Styles for these slots are project-owned. `openpress-create-slide` may create or edit local CSS, CSS modules, Tailwind classes, or theme tokens inside the Press folder. Core should not own slide typography classes. Agents may write the visual scale for each generated deck, but they must keep it attached to local layout and UI roles and verify fixed-canvas fit.

## Slot Boundary Decision Model

Agents need to decide which parts of a layout are slots and which parts are fixed layout structure. Named subcomponents are not automatically slots; some are layout primitives that protect a repeated structure.

Use this boundary:

| Region type | Agent behavior | Example |
| --- | --- | --- |
| Fixed layout frame | Layout owns it. Do not expose as content props unless it is a real variant. | safe area, grid, gutters, dividers, background panel, footer chrome |
| Formal text slot | Expose as Text-backed compound component. | `TitledContentSlide.Title`, `ChapterOpenerSlide.Lead` |
| Fixed local primitive | Expose a small component when the layout is fixed but repeated. It may accept small scalar values if it renders them through `Text`. | `ChapterOpenerSlide.ListItem` |
| Flexible content slot | Expose as children or a named slot. Do not force a data shape. | `TitledContentSlide.Content`, `ChapterOpenerSlide.Visual`, `TwoColumnSlide.Left` |
| Data adapter | Avoid as the default authoring API. | `items`, `metrics`, `steps`, `blocks` arrays |

The agent should ask what the user is likely to replace later. If the user may replace the whole region with a diagram, chart, screenshot, or custom JSX, it should be a flexible slot. If the user will usually only change the words inside the same repeated structure, it can be a fixed local primitive. If the region is purely deck chrome or layout scaffolding, keep it inside the layout.

## Slot Frame Layout Best Practices

Slide layouts should preserve flexible content slots. The goal is to give agents and users a stable frame for common layouts while keeping the actual content editable as JSX.

### `TitledContentSlide`

This is the most common slide shape: title area plus a lower content slot.

```tsx
<TitledContentSlide id="process-map" title="從題目到 Demo 的工作流程">
  <TitledContentSlide.Eyebrow objectId="eyebrow">
    Development Process
  </TitledContentSlide.Eyebrow>

  <TitledContentSlide.Title objectId="title">
    從題目到 Demo 的工作流程
  </TitledContentSlide.Title>

  <TitledContentSlide.Content>
    <div className="process-map">
      <article className="process-card">
        <Text as="span" objectId="process-01-number">01</Text>
        <Text as="h3" objectId="process-01-title">Discovery</Text>
        <Text as="p" objectId="process-01-body">釐清題目情境、使用者與 feature 範圍。</Text>
      </article>
      <article className="process-card">
        <Text as="span" objectId="process-02-number">02</Text>
        <Text as="h3" objectId="process-02-title">Planning</Text>
        <Text as="p" objectId="process-02-body">拆成 user story、acceptance criteria 與分工項目。</Text>
      </article>
    </div>
  </TitledContentSlide.Content>
</TitledContentSlide>
```

The layout owns the title layout and safe area. Its formal heading slots wrap `Text`. The lower content remains arbitrary JSX: process cards, charts, image grids, tables, or diagrams. Do not replace this slot with `steps`, `items`, or `cards` props.

### `ChapterOpenerSlide`

Chapter openers are a distinct layout. They usually combine chapter identity, a section list, and a visual panel.

Recommended compound-slot shape:

```tsx
<ChapterOpenerSlide id="chapter-topic-intro" title="題目介紹">
  <ChapterOpenerSlide.Chapter objectId="chapter">
    Chapter 01
  </ChapterOpenerSlide.Chapter>

  <ChapterOpenerSlide.Title objectId="title">
    01 題目介紹
  </ChapterOpenerSlide.Title>

  <ChapterOpenerSlide.Lead objectId="lead">
    先說明系統背景、組員角色，以及本次報告聚焦的考後流程缺口。
  </ChapterOpenerSlide.Lead>

  <ChapterOpenerSlide.SectionList>
    <ChapterOpenerSlide.ListItem id="topic-background" marker="01.1">
      題目背景與目標
    </ChapterOpenerSlide.ListItem>
    <ChapterOpenerSlide.ListItem id="feature-scope" marker="01.2">
      Feature 實務範圍
    </ChapterOpenerSlide.ListItem>
    <ChapterOpenerSlide.ListItem id="demo-output" marker="01.3">
      Demo 與報告輸出
    </ChapterOpenerSlide.ListItem>
  </ChapterOpenerSlide.SectionList>

  <ChapterOpenerSlide.Visual>
    <ProcessDiagram />
  </ChapterOpenerSlide.Visual>
</ChapterOpenerSlide>
```

This keeps the right-side frame flexible. It can hold a diagram, photograph, map, chart, product screenshot, or custom illustration. The chapter, title, and lead slots remain editable because they are Text-backed.

The section list is more fixed than the visual panel. `SectionList` owns the opener's list layout. `ListItem` owns the marker/title row and may derive object ids such as `topic-background-marker` and `topic-background-title`. This is acceptable because each item is still explicit JSX. Do not replace the whole list with `sections={[...]}` as the default output.

Compound slots are appropriate when a layout has a few named regions. They should stay shallow and obvious. If a slot component starts accumulating broad content props, the agent should re-evaluate the boundary: either keep it as fixed layout, expose a narrower local primitive, or move the content back into JSX.

### Layout And UI Classification

Standard slide layouts should be slot frames:

| Layout | Fixed by layout | Supplied by children or named slots |
| --- | --- | --- |
| `TitleSlide` | title-safe layout and cover variant | subtitle, metadata, visual accents |
| `TitledContentSlide` | heading area and content safe area | all body content |
| `ChapterOpenerSlide` | chapter identity layout and section-list structure | visual panel and any non-standard section content |
| `TwoColumnSlide` | two-column grid | both columns |
| `FullImageSlide` | full-bleed or framed media region | image, caption, overlay copy |
| `QuoteSlide` | quote scale and placement | quote text, attribution, optional visual |

Layouts are page-level presets, not data adapters. UI primitives are smaller reusable content blocks that can appear inside any layout:

| UI primitive | Responsibility |
| --- | --- |
| `Text` | Editable text foundation and object id wrapper |
| `Card`, `Badge`, `Callout` | Small reusable content treatments |
| `KpiCard` | Metric block for dashboards or evidence slides |
| `ImageFrame` | Framed media with optional caption |
| `QuoteBlock` | Quote text and attribution |
| `Timeline` | Reusable process or sequence visualization |
| `CompareTable` | Comparison matrix or before/after table |

Applied examples:

- A chapter opener slide belongs in `layouts/chapter-opener-slide.tsx`. `ChapterOpenerSlide.SectionList` and `ChapterOpenerSlide.ListItem` are layout-local primitives because the opener section list is part of that page structure. A right-side process illustration, chart, or screenshot should be supplied through `ChapterOpenerSlide.Visual`; if it becomes reusable, put it in `ui/`.
- A titled process slide belongs in `layouts/titled-content-slide.tsx`. The layout owns the title and content regions. A `Timeline` inside the content region belongs in `ui/timeline.tsx` because it can be reused inside other slide layouts.

## Object And Comment Model

Slide text that should be editable or commentable should use `Text` in visible JSX, a Text-backed compound slot, or a local component that wraps `Text`. It should not live in string fields inside arrays.

Good:

```tsx
<Text as="h3" objectId="metric-01-title">可驗收成果</Text>

<TitledContentSlide.Title objectId="title">
  從題目到 Demo 的工作流程
</TitledContentSlide.Title>
```

Weak:

```tsx
metrics={[{ title: "可驗收成果" }]}
```

This keeps OpenPress's source mapping, object entity model, inspector comments, inline edit, and text replacement tools aligned with what the user sees on the slide. Plain JSX text can still be used for decorative chrome or generated labels that are not intended to be edited.

## Theme And Media Ownership

Per-Press theme and media are standard. Shared folders are optional.

```text
press/slide/theme/       # slide-specific visual system
press/slide/media/       # slide-specific assets
press/shared/theme/      # optional shared brand layer
press/shared/media/      # optional shared assets
```

The engine should support per-Press theme and media resolution as part of the folder convention. Shared resources can be imported or referenced deliberately by each Press, but they should not be the only supported location.

## Skill Changes

### `openpress-create-pages`

The pages skill should create or update only one Press folder:

```bash
openpress init <press-name> --type pages
```

It should generate:

- `press/<folder>/press.tsx`
- `press/<folder>/chapters/`
- `press/<folder>/components/`
- `press/<folder>/theme/`
- `press/<folder>/media/`

### `openpress-create-slide`

The slide skill should own slide creation, slide editing, and slide layout/template authoring. It should not only scaffold files; it should guide the agent through the reasoning needed to choose structure, slots, visual hierarchy, density, reusable layout boundaries, and reusable UI primitives.

Trigger this skill when the user asks to:

- create a new slide Press or presentation;
- add, remove, reorder, or edit slides;
- make or revise slide layouts/templates;
- convert a slide from data props to slot-frame JSX;
- adjust slide visual structure, density, chrome, folio, or reusable layout components;
- interpret "this slide", "this template", "the opener", "the two-column page", or similar slide-specific references.

Route non-slide document creation to `openpress-create-pages`. Route CLI lifecycle, upgrade, migration, render, export, deploy, and doctor work to `openpress`.

For a new slide Press, it should create or update only one Press folder:

```bash
openpress init <press-name> --type slides
```

It should generate:

- `press/<folder>/press.tsx`
- `press/<folder>/components/DeckSlide.tsx`
- `press/<folder>/ui/*.tsx`
- `press/<folder>/layouts/*.tsx`
- `press/<folder>/theme/`
- `press/<folder>/media/`

It should ask for topic, audience, page count, density, visual direction, and assets, but it should write actual slide content as JSX children rather than array props. It may write or revise local style files for the generated deck; slide styling is part of the project-owned layout and UI surface, not a hidden core concern.

### Slide Skill Guidance Model

The slide skill should guide the agent through these decisions before editing source:

1. Identify the operation: create Press, add slide, edit slide, make layout/template, revise layout/template, or migrate an existing props-heavy layout.
2. Identify the slide role: title, chapter opener, titled content, two-column, full image, quote, agenda, process, comparison, closing, or custom.
3. Pick the right base layer:
   - use core `Slide` only for a blank slide primitive;
   - use project `DeckSlide` for deck chrome, watermark, footer, folio, and safe area;
   - use `layouts/*` for full-slide slot-frame layout presets;
   - use `ui/*` for reusable content primitives inside layout slots.
4. Decide which parts are formal layout metadata and which parts must stay editable JSX:
   - metadata props may include `id`, navigation `title`, `chapter`, `variant`, and `className`;
   - visible title, eyebrow, description, lead, caption, and labels should use Text-backed compound slots;
   - fixed repeated structures may use local primitives such as `ChapterOpenerSlide.ListItem`;
   - content, cards, lists, diagrams, images, and right-side frames should use children or named slots.
5. Decide the slot boundary before inventing props:
   - fixed layout frame stays inside `layouts/*`;
   - formal text becomes Text-backed compound slots;
   - repeated fixed rows become narrow local primitives;
   - replaceable regions become children or named slots;
   - reusable content blocks become `ui/*` primitives when they can appear in multiple layouts.
6. Use stable semantic slide ids. Reordering must not require O(n) key rewrites.
7. Prefer `Text` or Text-backed compound slots for editable visible text.
8. Write project-owned styles for layout and UI roles when needed. Do not push typography or deck-specific styling into core.
9. Use `PageFolio` for folio rendering. Never pass `pageNumber` or `totalPages`.
10. Verify fixed-canvas fit: heading size, line count, content budget, and whether the idea should split into multiple slides.

Layout/template-making guidance:

- Start from one concrete slide the user wants.
- Name the invariant layout frame.
- Decide the boundary between fixed layout, formal text slots, fixed local primitives, and flexible content slots.
- Keep flexible regions as children or named slots.
- Use compound components for formal text slots such as title, eyebrow, description, chapter, lead, quote, and attribution.
- Use local primitives for narrow fixed structures such as `ChapterOpenerSlide.SectionList` and `ChapterOpenerSlide.ListItem`.
- Extract `ui/*` primitives only when the content block is reusable across multiple layouts or makes a slot easier to read.
- Promote repeated visual structure into CSS/classes before promoting content into props.
- Add a layout only when it will be reused or clarifies a common slide role.
- Do not make a layout just to avoid writing JSX.

Editing guidance:

- If changing slide order, move JSX blocks and keep semantic `id` values unchanged.
- If inserting a slide, choose a new semantic `id`.
- If editing content, edit visible JSX, `Text` nodes, and Text-backed compound slots first.
- If a current layout hides content inside arrays, prefer migrating that slide to slot-frame JSX before making further content edits.

## Migration Strategy

Existing projects must migrate to `press/*/press.tsx`. There is no root-entry compatibility path in this version.

Migration from root-composed slides:

1. Move each artifact into `press/<folder>/press.tsx`.
2. Move deck-level wrappers into `press/<folder>/components/`.
3. Move reusable content primitives into `press/<folder>/ui/`.
4. Move slide-level layout components into `press/<folder>/layouts/`.
5. Replace manual `pageNumber` and `totalPages` props with `PageFolio`.
6. Replace visible string props with Text-backed compound slots when the text needs inline editing or comment targeting.
7. Replace data-prop layouts with children-composed JSX where source editing matters.
8. Move artifact-specific media and theme files into the Press folder.
9. Delete the root Press entry; no compatibility path remains.

## Acceptance Criteria

- A new pages Press can be created with `openpress init report --type pages`.
- A new slides Press can be created with `openpress init slide --type slides`.
- A workspace with one or more `press/*/press.tsx` files exports correctly.
- `press/shared/` is ignored as a Press entry.
- Slide layouts generated by `openpress-create-slide` do not require `items`, `metrics`, `steps`, `pageNumber`, or `totalPages`.
- Generated slide ids are stable semantic ids such as `cover`, `agenda`, or `problem-context`, not order-carrying ids such as `slide-01`.
- The default slide scaffold includes `components/DeckSlide.tsx`, not `SlideFrame.tsx`, `SlideHeader.tsx`, or `SlideFooter.tsx`.
- The default slide scaffold includes `ui/` for reusable content primitives and `layouts/` for full-slide layout components.
- The default slide scaffold does not use `templates/` as the source folder name.
- Standard layouts expose content through children or named slots.
- Standard layouts expose formal visible text through Text-backed compound slots such as `TitledContentSlide.Title` and `TitledContentSlide.Description`.
- Generated slide Press entries compose visible content inline inside layout components instead of hiding every slide behind one empty one-off component.
- Chapter opener layouts may expose fixed local primitives such as `ChapterOpenerSlide.SectionList` and `ChapterOpenerSlide.ListItem` when the repeated structure is part of the opener layout.
- Slide create, edit, layout authoring, and template-making requests route to the slide skill.
- The slide skill contains guidance for choosing between `Slide`, `DeckSlide`, `layouts/*`, and `ui/*`.
- The slide skill teaches agents to classify regions as fixed layout, formal text slots, fixed local primitives, flexible content slots, or data adapters before introducing props.
- The slide skill tells agents they may write project-owned layout/UI styles and theme files, while core stays visually unopinionated.
- The slide skill instructs agents to migrate props-heavy layouts toward slot-frame JSX when editing visible content.
- Slide page numbers render through `PageFolio`.
- Core `Slide` is a thin wrapper around `Frame` and does not own visual styling.
- Root-only workspaces do not render; they must migrate to `press/*/press.tsx`.

## Decisions Captured

- Use `shared`, not `_shared`.
- Root Press entries are not supported in the new convention.
- Press order is not a design priority in this version.
- `type` has only two supported values: `pages` and `slides`.
- Slide source uses `ui/` for reusable content primitives and `layouts/` for full-slide layout components.
- `templates/` is not the default slide source folder; "template" remains skill/generation vocabulary.
- Generated `press.tsx` should use `Press > LayoutSlide > inline content`; one component per individual slide is not the default.
- Slide authoring should be shadcn-style project source, not a core component library.
- A minimal core `Slide` primitive is useful, but only as a semantic wrapper around `Frame`.
- The author-facing slide identity prop is `id`; it maps to `Frame`'s internal `frameKey`.
- Do not generate order-based slide ids by default.
- Use `DeckSlide` for the deck-local blank slide with watermark/chrome/folio.
- `SlideHeader`, `SlideFooter`, and `SlideSection` are optional extractions, not default scaffold files.
- Layout best practices should preserve slot-frame components such as `TitledContentSlide` and `ChapterOpenerSlide`.
- Visible layout text should use role-based, Text-backed compound slots rather than an `H1` / `H2` / `H3` component API.
- Agents must decide the boundary between fixed layout structure, formal text slots, fixed local primitives, and flexible content slots before adding layout props.
- `ChapterOpenerSlide.SectionList` and `ChapterOpenerSlide.ListItem` are valid fixed local primitives because opener section lists are part of a relatively fixed layout.
- Agents may author local layout/UI styles and theme files for each slide Press; core remains visually unopinionated.
- `openpress-create-slide` is also the slide edit and slide template-making skill, not only the create workflow.
