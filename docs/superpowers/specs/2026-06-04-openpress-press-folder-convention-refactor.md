# OpenPress Press Folder Convention Refactor Spec

## Summary

OpenPress should move from a root-entry composition model to a filesystem-first Press folder convention.

The next version should treat every user artifact as a self-contained folder under `press/<folder>/`. Each folder owns its own `press.tsx`, local components, theme, and media. The engine discovers those Press folders, loads each folder's exported Press component, and builds the Workspace manifest from that collection.

This refactor is motivated by the create-pages / create-slide skill split and by dogfood from a generated slide project. The current generated slide output works, but it tends to produce props-heavy template DSLs such as `items`, `metrics`, `steps`, `pageNumber`, and `totalPages`. That shape is fast for first generation but weak for OpenPress's long-term source editing, comment targeting, and agent follow-up work.

The new slide authoring model should be closer to shadcn-style copied project components: minimal props, explicit JSX children, local ownership, and easy editing by humans or agents.

## Goals

- Make `press/<folder>/` the unit of authoring, editing, export, and agent reasoning.
- Let `openpress init --folder <name> --type pages|slides` scaffold a single Press folder.
- Limit supported Press types to `pages` and `slides`.
- Remove the need for a required root `press/index.tsx` in the default path.
- Keep shared resources possible through `press/shared/`, but make shared resources optional rather than the default owner of all theme and media.
- Make slide generation produce React source that is directly readable and editable, not a props DSL.
- Provide a thin core `Slide` primitive that wraps `Frame` for slide semantics.
- Keep design-heavy slide components project-owned and generated by `openpress-create-slide`, not hidden inside core.

## Non-Goals

- Do not port the `open-slide` runtime into OpenPress.
- Do not add a third Press type such as `social`; social-card outputs remain either `pages` or a future separate decision.
- Do not design a `Deck` abstraction in this version.
- Do not require root-level `press/index.tsx` ordering overrides in this version.
- Do not create a large slide component library inside `@open-press/core`.
- Do not make slide templates accept large data props such as `items`, `metrics`, or `steps` as the primary authoring style.

## Current Problems

### Root Entry Owns Too Much

The current multi-Press model relies on `press/index.tsx` to mount a `Workspace` and compose every `Press`. In a generated project, that often becomes:

```tsx
import { Press, Workspace } from "@open-press/core";
import { config, qjudgeScrumFeatureReviewFrames } from "./qjudge-scrum-feature-review";

export default function OpenPressWorkspace() {
  return (
    <Workspace name="QJudge Scrum Feature Review">
      <Press slug="qjudge-scrum-feature-review" title={config.title} type="slides" page={config.page}>
        {qjudgeScrumFeatureReviewFrames()}
      </Press>
    </Workspace>
  );
}
```

This keeps the real Press metadata outside the folder that owns the deck. The deck folder exports frames and config, while the root entry remains the actual Press registry. That makes the root file a coordination bottleneck and forces create skills to modify shared project state even when adding one artifact.

### Generated Slide Templates Become Data DSLs

The generated slide project uses templates like:

```tsx
<AgendaSlide
  pageNumber={2}
  totalPages={totalPages}
  items={[
    { number: "01", title: "題目介紹", note: "交代 QJudge feature 的情境、使用者與報告範圍。" },
    { number: "02", title: "敏捷會議計劃", note: "規劃會議節奏、角色分工與每次討論的產出。" },
  ]}
/>
```

This creates several issues:

- The visible slide content is stored inside object arrays instead of slide JSX.
- Follow-up edits need to modify data objects rather than the exact rendered structure.
- `items.map()` output is weaker for source mapping and comment targeting.
- Template components tend to accumulate more props over time.
- Page numbers are hardcoded into props, so slide insertion or reordering causes unrelated edits.

OpenPress should bias toward explicit source because source is the durable collaboration surface.

## Reference: open-slide

The `1weiho/open-slide` project is useful as a reference for slide authoring philosophy:

- A slide page is arbitrary React code.
- The canvas is fixed at 1920 x 1080.
- Agent rules are shipped with the scaffold.
- A slide has a local folder and local assets.

OpenPress should adopt the "slides are React source, not a constrained DSL" principle. It should not copy open-slide's one-large-file rule. OpenPress needs long-term document maintenance, comments, source editing, PDF/image export, and folder-level ownership, so multi-file Press folders are a better fit.

## Proposed Project Layout

### Default Workspace

```text
press/
├── shared/
│   ├── data.ts
│   ├── components/
│   ├── media/
│   └── theme/
├── report/
│   ├── press.tsx
│   ├── chapters/
│   ├── components/
│   ├── theme/
│   └── media/
└── slide/
    ├── press.tsx
    ├── components/
    ├── templates/
    ├── theme/
    └── media/
```

`press/shared/` is optional. It exists only when multiple Press folders need common assets, shared facts, brand tokens, or reusable components.

Each Press folder is the default source of truth for its own artifact. In the common case, an agent can add or modify a Press without touching sibling folders.

### Required Per-Press Files

Every `press/<folder>/` must contain:

```text
press/<folder>/
├── press.tsx
├── components/
├── theme/
└── media/
```

Additional required files depend on Press type:

```text
press/<folder>/chapters/     # required for type="pages"
press/<folder>/templates/    # required for type="slides"
```

The folder name is the default slug. The `Press` component may omit `slug` only if the engine injects or validates it from the folder name. The preferred explicit form is:

```tsx
<Press slug="slide" title="..." type="slides" page="slide-16-9">
  ...
</Press>
```

## Filesystem Discovery

The engine should discover Press entries by scanning:

```text
press/*/press.tsx
```

Only one convention should be recognized in this version. Do not also support `index.tsx`, `deck.tsx`, `document.tsx`, or `page.tsx` as equivalent Press entries.

Discovery rules:

- Ignore `press/shared/`.
- Ignore dot folders and generated output.
- Sort folders by stable filesystem name for deterministic output.
- Load each `press/<folder>/press.tsx` default export.
- The default export must be an inert component that returns one `<Press>`.
- The folder name becomes the default slug and must match the explicit `slug` prop if one is present.
- The engine builds a Workspace internally from discovered Press entries.

The root `press/index.tsx` is not required in this version. Ordering is not a first-class concern yet.

## CLI Init Contract

`openpress init` should become folder-aware:

```bash
openpress init --folder report --type pages
openpress init --folder slide --type slides
```

Behavior:

- Create the target workspace if it does not exist.
- Create `press/<folder>/` if it does not exist.
- Scaffold the required folder structure for the requested type.
- Refuse to overwrite an existing non-empty Press folder unless an explicit future flag is added.
- Keep lifecycle operations such as upgrade and migrate under the general `openpress` workflow.

Supported `--type` values:

```text
pages
slides
```

There is no `social` Press type in this version.

## Pages Folder Contract

Default pages layout:

```text
press/report/
├── press.tsx
├── chapters/
├── components/
├── theme/
└── media/
```

Default `press.tsx` shape:

```tsx
import { Press } from "@open-press/core";
import { mdxSource } from "@open-press/core/mdx";
import { Sections, Toc } from "@open-press/core/manuscript";

export default function ReportPress() {
  return (
    <Press
      slug="report"
      title="Report"
      type="pages"
      page="a4"
      sources={[
        mdxSource({ id: "report", preset: "section-folders", root: "report/chapters" }),
      ]}
    >
      <Toc source="report" maxLevel={2} />
      <Sections source="report" />
    </Press>
  );
}
```

Pages continue to use MDX source registration. The create-pages skill owns cover, back-cover, chapter opener, TOC depth, theme, and manuscript structure.

## Slides Folder Contract

Default slides layout:

```text
press/slide/
├── press.tsx
├── components/
│   ├── SlideFrame.tsx
│   ├── SlideHeader.tsx
│   └── SlideFooter.tsx
├── templates/
│   ├── TitleSlide.tsx
│   ├── AgendaSlide.tsx
│   ├── ContentSlide.tsx
│   └── ClosingSlide.tsx
├── theme/
└── media/
```

Default `press.tsx` shape:

```tsx
import { Press } from "@open-press/core";
import { AgendaSlide } from "./templates/AgendaSlide";
import { ClosingSlide } from "./templates/ClosingSlide";
import { ContentSlide } from "./templates/ContentSlide";
import { TitleSlide } from "./templates/TitleSlide";

export default function SlidePress() {
  return (
    <Press slug="slide" title="Slide Deck" type="slides" page="slide-16-9">
      <TitleSlide frameKey="slide-01" />
      <AgendaSlide frameKey="slide-02" />
      <ContentSlide frameKey="slide-03" />
      <ClosingSlide frameKey="slide-04" />
    </Press>
  );
}
```

There is no `Deck` abstraction in this version. The Press entry composes slides directly.

## Core `Slide` Primitive

Core should provide a thin `Slide` component that wraps `Frame`:

```tsx
import { Slide } from "@open-press/core";

<Slide frameKey="slide-01" title="Agenda">
  ...
</Slide>
```

Expected behavior:

```tsx
<Frame
  frameKey={frameKey}
  role="canvas.slide"
  chrome={false}
  data-page-title={title}
>
  {children}
</Frame>
```

`Slide` should not own visual design. It should not include theme, headers, footers, grids, logos, or layout opinions.

Minimal props:

- `frameKey`
- `title?`
- `role?`, defaulting to `canvas.slide`
- `className?`
- `children`

The purpose is to prevent agents from repeatedly remembering the `Frame` slide contract while keeping all design choices in project-owned files.

## Shadcn-Style Slide Components

`openpress-create-slide` should generate project-owned components that the user can edit directly. These components should feel like shadcn components: copied into the project, explicit, and easy to modify.

Recommended local component responsibilities:

| Component | Responsibility |
| --- | --- |
| `SlideFrame` | Visual shell around core `Slide`; owns background, surface class, shared layout slots. |
| `SlideHeader` | Optional fixed header with deck label, section label, logo, or metadata. |
| `SlideFooter` | Optional fixed footer with `PageFolio`, source label, or progress marker. |
| `SlideSection` | Reusable spacing container for title/body blocks. |

`SlideFrame` should use `PageFolio` internally or through `SlideFooter`. Agents should not pass `pageNumber` or `totalPages`.

## Slide Template Style

Slide templates should prefer children composition over data props.

Preferred:

```tsx
<AgendaSlide frameKey="slide-02">
  <SlideSection>
    <Text as="p" objectId="agenda-eyebrow">Agenda</Text>
    <Text as="h2" objectId="agenda-title">本次報告結構</Text>
  </SlideSection>

  <ol className="agenda-list">
    <li>
      <span>01</span>
      <div>
        <Text as="h3" objectId="agenda-01-title">題目介紹</Text>
        <Text as="p" objectId="agenda-01-note">交代情境、使用者與報告範圍。</Text>
      </div>
    </li>
  </ol>
</AgendaSlide>
```

Avoid:

```tsx
<AgendaSlide
  items={[
    { number: "01", title: "題目介紹", note: "交代情境、使用者與報告範圍。" },
  ]}
/>
```

Allowed props should be structural, not content-heavy:

- `frameKey`
- `title?` for `data-page-title`
- `variant?` for explicit layout variants
- `className?`
- `children`

Avoid defaulting to props such as:

- `items`
- `metrics`
- `steps`
- `blocks`
- `pageNumber`
- `totalPages`
- `logo`
- `footerLabel`
- `showFolio`

If a slide genuinely needs repeated data, write the repeated JSX in the slide source first. Introduce a small local helper only when repetition is high and the helper improves readability without hiding source ownership.

## Object And Comment Model

Slide text that should be editable or commentable should use `Text` in visible JSX, not string fields inside arrays.

Good:

```tsx
<Text as="h3" objectId="metric-01-title">可驗收成果</Text>
```

Weak:

```tsx
metrics={[{ title: "可驗收成果" }]}
```

This keeps OpenPress's source mapping, object entity model, inspector comments, and text replacement tools aligned with what the user sees on the slide.

## Theme And Media Ownership

Per-Press theme and media are standard. Shared folders are optional.

```text
press/slide/theme/       # slide-specific visual system
press/slide/media/       # slide-specific assets
press/shared/theme/      # optional shared brand layer
press/shared/media/      # optional shared assets
```

The engine should support per-Press theme and media resolution as part of the folder convention. Shared resources can be imported or referenced deliberately by each Press, but they should not be the only supported location.

## Skill Changes

### `openpress-create-pages`

The pages skill should create or update only one Press folder:

```bash
openpress init --folder <folder> --type pages
```

It should generate:

- `press/<folder>/press.tsx`
- `press/<folder>/chapters/`
- `press/<folder>/components/`
- `press/<folder>/theme/`
- `press/<folder>/media/`

### `openpress-create-slide`

The slide skill should create or update only one Press folder:

```bash
openpress init --folder <folder> --type slides
```

It should generate:

- `press/<folder>/press.tsx`
- `press/<folder>/components/SlideFrame.tsx`
- `press/<folder>/components/SlideHeader.tsx`
- `press/<folder>/components/SlideFooter.tsx`
- `press/<folder>/templates/*.tsx`
- `press/<folder>/theme/`
- `press/<folder>/media/`

It should ask for topic, audience, page count, density, visual direction, and assets, but it should write actual slide content as JSX children rather than array props.

## Migration Strategy

Existing projects can keep `press/index.tsx` during a compatibility window. New projects should use folder discovery by default.

Migration from root-composed slides:

1. Move each artifact into `press/<folder>/press.tsx`.
2. Move local templates into `press/<folder>/templates/`.
3. Move local components into `press/<folder>/components/`.
4. Replace manual `pageNumber` and `totalPages` props with `PageFolio`.
5. Replace data-prop templates with children-composed JSX where source editing matters.
6. Move artifact-specific media and theme files into the Press folder.
7. Remove root `press/index.tsx` once filesystem discovery is active.

## Implementation Phases

### Phase 1: Spec And Scaffold Contract

- Document Press Folder Convention.
- Update create-pages and create-slide docs.
- Update skills to describe `press/<folder>/press.tsx`.
- Keep runtime behavior unchanged until engine support lands.

### Phase 2: Engine Discovery

- Add discovery for `press/*/press.tsx`.
- Build an internal Workspace from discovered Press entries.
- Validate folder slug and `Press slug` consistency.
- Preserve existing root `press/index.tsx` compatibility for older projects.

### Phase 3: CLI Init

- Add `openpress init --folder <name> --type pages|slides`.
- Scaffold type-specific folder structures.
- Refuse unsafe overwrites.

### Phase 4: Slide Primitive And Skill Output

- Add core `Slide`.
- Update create-slide starter output to use `SlideFrame` wrapping `Slide`.
- Update templates to use children composition and `PageFolio`.

### Phase 5: Migration Tooling

- Add validation warnings for root-composed Press trees in new workspaces.
- Add migration guidance or a dry-run command for moving root-composed artifacts into folder convention.

## Acceptance Criteria

- A new pages Press can be created with `openpress init --folder report --type pages`.
- A new slides Press can be created with `openpress init --folder slide --type slides`.
- A workspace with no `press/index.tsx` and one or more `press/*/press.tsx` files exports correctly.
- `press/shared/` is ignored as a Press entry.
- Slide templates generated by `openpress-create-slide` do not require `items`, `metrics`, `steps`, `pageNumber`, or `totalPages`.
- Slide page numbers render through `PageFolio`.
- Core `Slide` is a thin wrapper around `Frame` and does not own visual styling.
- Existing root-entry workspaces continue to render during the compatibility window.

## Decisions Captured

- Use `shared`, not `_shared`.
- `press/index.tsx` is not part of the default new-project convention.
- Press order is not a design priority in this version.
- `type` has only two supported values: `pages` and `slides`.
- Slide `templates/` is part of the standard slides scaffold.
- Slide authoring should be shadcn-style project source, not a core component library.
- A minimal core `Slide` primitive is useful, but only as a semantic wrapper around `Frame`.
